# TensorGuard Vue 产品演示工作台设计

## 1. 状态

- 设计状态：已经用户确认。
- 目标产物：基于 Vue 3 的 TensorGuard 前端初版。
- 核心约束：重写展示层，保持现有后端 API、实验流程、数据路径和判定规则不变。

## 2. 目标与非目标

### 2.1 目标

1. 将当前 `HTML/CSS/JS` 展示层迁移到 Vue 3，使三个主页面拥有稳定的组件边界和一致视觉系统。
2. 面向作品赛现场演示，让评委快速理解 API 覆盖、自动化检测过程和 Bug 证据。
3. 保留 API 搜索、单 API 完整运行、实时轮询、Results 审查、候选收录、Bug 复现和报告生成的全部能力。
4. 所有环境、API 数量、模型名称、GPU 信息和实验指标均来自实时接口，不把设计稿数据写死到正式前端。

### 2.2 非目标

- 不修改 Qwen、InCoder、Driver 和 Bug 复现子进程的实现。
- 不修改 `Results/`、`demo_runs/`、`reports/` 和候选库的存储规则。
- 不修改已确认 Bug 的判定结果，不用前端展示逻辑替代后端证据分类。
- 不在总览页生成不存在的历史趋势或虚构运行数据。

## 3. 技术架构

### 3.1 前端栈

- Vue 3 Single-File Components，使用 Composition API 和 TypeScript。
- Vite 负责开发、测试和生产构建。
- Arco Design Vue 仅用于选择器、搜索、表格、抽屉、提示和对话框等基础交互控件；三个页面的主布局与品牌视觉使用自定义组件。
- Apache ECharts 负责 Qwen、InCoder、Driver 和 GPU 采样曲线。
- Lucide Vue Next 负责导航、状态和操作图标。

### 3.2 目录与构建

- Vue 源码位于 `webapp/frontend/`。
- Vite 生产输出位于 `webapp/static/`，保持 `webapp/server.py` 的 `STATIC_ROOT` 不变。
- 开发模式下 Vite 将 `/api` 代理到 Python 服务；生产模式下使用同源相对路径。
- 部署时仍然只需运行：

```bash
python webapp/server.py --host 0.0.0.0 --port 42173
```

- Vue 构建完成后不需要服务器常驻 Node 进程。

### 3.3 前端分层

1. `services/`：封装现有 HTTP 路由，不改变请求与响应结构。
2. `composables/`：管理环境、API 任务、阶段跟随、Results 审查、Bug 列表、复现任务和报告状态。
3. `components/`：只负责可复用交互和展示，不直接组装后端 URL。
4. `views/`：组装系统总览、单 API 运行和 Bug 复现三个页面。
5. `types/`：定义环境、任务、阶段采样、Results、Bug 记录、复现证据和报告结构。

## 4. 导航与通用视觉

- 三个页面使用相同的顶部导航、最大内容宽度、间距标准和响应式断点。
- 导航包含“系统总览”“单 API 运行”“Bug 复现”三项，不在页面标题区重复放置相同跳转按钮。
- 顶部右侧展示实时环境状态，点击后打开环境抽屉。
- 主视觉使用明亮冷灰背景、白色表面和钴蓝主操作色；成功、警告、异常分别使用绿色、琳珀色和红色。
- 页面高度不写死，优先在常用桌面视口展示主要内容；内容放不下时使用正常纵向滚动，不为强制一屏而删除功能。
- 宽度不低于 720px 时优先保持主次双列；更窄时改为单列，但主导航始终可见。

## 5. 系统总览

### 5.1 信息架构

1. 页面标题为“框架安全检测概览”。
2. 首要区域展示 API 覆盖基线：总数、PyTorch 数量、TensorFlow 数量及比例。
3. 右侧展示已确认问题数量，仅统计可被现有确认库解析的记录。
4. 中部展示约束库状态、测试框架数和执行环境。
5. 底部用有方向的节点和箭头表达“约束加载 → 种子生成 → 变异测试 → 差异检测 → 证据归档”。
6. 已确认证据列表显示真实 API、技术影响类型和证据状态，点击进入 Bug 复现页。

### 5.2 数据源

- `/api/overview`：API 数量、确认问题数量、约束库状态和概览快照。
- `/api/environment`：设备与框架环境。
- 概览数据加载失败时显示局部错误状态和重试按钮，其他页面仍可导航。

## 6. 单 API 运行

### 6.1 API 选择与运行

- 保留框架、API 搜索、运行模式、GPU 和运行按钮。
- API 列表继续来自 `/api/apis`；空搜索时可下拉浏览当前框架 API，输入关键词后实时过滤。
- 选中项必须有明确高亮反馈；切换框架后清空无效选择并重新查询。
- 选择 API 后请求 `/api/api-detail`，立即展示对应约束、最新任务状态和 Results 快照，不等待用户再次点击运行。
- 启动任务仍使用 `POST /api/run-api`，随后轮询 `/api/api-jobs/{job_id}`。

### 6.2 运行可视化

- 完整状态链保留五个节点：约束检查、Qwen 种子生成、InCoder 变异、CPU/GPU 差异检测、摘要生成。
- 主图表保留三个采样阶段：
  - Qwen：候选数和有效种子数。
  - InCoder：有效、异常、崩溃和超时数。
  - Driver：已检查程序、差异 Catch、崩溃和不稳定数。
- 三个阶段共用一个主图表，通过标签切换，不同时堆叠三张图。
- 尚未开始或跳过的阶段不可选；默认跟随当前活跃阶段，用户可回看已完成阶段并返回实时阶段。
- GPU 利用率和显存占比保留为独立监控图，不计入三个阶段图表。
- 图表仅绘制后端已采样的点，缺失值保持空缺，不补零或插值。

### 6.3 Results、日志与候选

- 实时摘要、Results 构成、实时日志和 GPU 监控保持与主图表列线对齐。
- Results 审查保留 `seed`、`valid`、`exception`、`crash`、`notarget`、`hangs`、`flaky` 七类。
- 用户可查看本次任务精确源码，并使用现有候选接口将文件加入证据候选库。
- 不为强制一屏展示而删除 Results 源码审查或日志；页面在需要时可纵向滚动。

## 7. Bug 复现

### 7.1 列表与问题结论

- 左侧保留“已确认／候选”来源标签、搜索框和可滚动问题列表。
- 正式界面不出现 `paper`、`PAPER-*` 等内部来源词；用户可见编号使用去前缀形式，如 `PT-004`。
- 详情顶部首先显示 API、技术影响类型、证据状态和当前环境复现结论。
- 初版不显示 High、Medium 等安全严重性，也不根据崩溃关键词自动推导 CVSS。界面只展示可由现有证据直接支持的“进程终止”“内存破坏”“内部断言”“静默错误”“CPU/GPU 不一致”等技术影响。
- 未来如果经人工完成 CVSS 评估，必须同时存储分数、完整向量、评估人和评估依据，否则不在界面展示安全严重性。
- “预期行为／实际行为”对照紧随结论展示，明确说明为什么该现象可能是框架缺陷。

### 7.2 复现和证据

- 最小复现代码与当前运行输出并列展示，明确区分归档内容和现场输出。
- CPU 和可用 GPU 继续作为独立执行配置，使用 `POST /api/confirmed-bugs/{id}/reproduce` 启动子进程。
- 轮询 `/api/repro-jobs/{job_id}` 显示返回码、信号、超时、实际设备、匹配规则和确定性判定。
- 复现结论只使用后端返回的 `evidence.verdict`；未命中规则的异常显示“需要人工审查”。
- 环境信息从本次复现任务的快照中读取，不写死 GPU 型号、CUDA 或框架版本。

### 7.3 报告

- 首次生成使用现有 `POST /api/repro-jobs/{job_id}/report`。
- 报告已存在时直接读取已归档 Markdown，不覆盖。
- 前端保留预览、Markdown 下载和打印／PDF 入口。
- 报告不依赖大模型，使用结构化任务数据和后端模板生成。

## 8. 状态、错误和边界情况

- 初始、加载、空数据、运行、成功、失败、超时和需审查状态均有明确展示。
- 后端返回非 2xx 时，优先显示结构化 `error`，并保留详细日志路径，不把后续阶段误标为已执行。
- 轮询会在完成、失败、组件卸载或用户离开页面时停止，避免重复定时器。
- 网络短暂失败保留当前已加载数据并提供重试，不立即清空整页。
- 没有 GPU 时显示真实“未检测到 GPU”，禁用 GPU 执行项，CPU 能力仍可使用。
- 没有图表采样时显示阶段化空状态，不显示伪造曲线。

## 9. 逻辑兼容映射

| 现有逻辑 | Vue 初版要求 |
| --- | --- |
| `state.selectedApi` | `useApiCatalog` 中的当前 API |
| `currentApiJob` / `apiPollTimer` | `useApiRun` 任务与可清理轮询 |
| `selectedApiStage` / `followLiveStage` | 三阶段图表选中与自动跟随 |
| `resultCategories` | 原七类 Results 及数量 |
| `selectedResultFile` | Results 精确文件预览 |
| `bugSource` / `selectedBugId` | 已确认／候选和当前问题 |
| `currentReproJob` / `reproPollTimer` | `useBugReplay` 复现任务与轮询 |
| `reportMarkdown` | 已归档报告预览与下载 |
| `stageDefinitions` | Qwen、InCoder、Driver 指标定义保持不变 |

## 10. 测试与验证

### 10.1 自动化测试

- Vitest 覆盖 API 服务、阶段计算、结果聚合、显示编号和报告不覆盖状态。
- Vue Test Utils 覆盖 API 搜索、选中高亮、阶段禁用、活跃阶段跟随、Results 分类和 Bug 来源切换。
- 保留现有 Python 后端测试，确认 Vue 迁移未修改服务行为。
- 增加前后端契约 fixture，覆盖成功、失败、无 GPU、空采样、候选和报告已存在。

### 10.2 浏览器验证

- 启动真实 Python 服务，验证三个页面的导航、数据加载和空状态。
- 使用可控 fixture 验证运行中、完成、失败和复现成功视图，不要求每次视觉测试真正加载大模型。
- 在 1366×768、1440×900 和约 760px 宽度下检查不重叠、不溢出、导航可见和图表尺寸稳定。
- 验证三个页面的顶部导航、内容宽度和标题比例一致，允许内容较多的页面正常纵向滚动。
- 验证打印／PDF 报告不包含导航和操作按钮。

## 11. 完成标准

1. 正式页面默认使用已确认的明亮产品演示风格和顶部导航。
2. 三个页面都能从真实接口获取数据，不包含设计稿写死数值。
3. 现有 API 搜索、任务轮询、阶段图表、Results 审查、候选收录、Bug 复现和报告规则均通过回归测试。
4. Python 服务启动命令和现有 API 路由不变。
5. 生产构建、前端测试、后端测试和浏览器视觉检查全部通过。
