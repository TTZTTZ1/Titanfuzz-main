## Purpose
- Detect stuck workers using countdown timers
- Automatically terminate workers that exceed specified time limits
- Enable automatic retry of failed worker groups

### **Timer Server** (Agent-side)
- Monitors active timers and expires them
- Terminates workers with expired timers
- Runs as a daemon process alongside the agent

## Provided Implementations

1. **LocalTimerServer/LocalTimerClient**
   - Uses `multiprocessing.Queue` for communication
   - Designed for subprocess workers on same host

2. **FileTimerServer/FileTimerClient**
   - Uses named pipes (FIFO) for communication
   - Multiple clients can connect to same server

## Key Features
- **Automatic cleanup**: Stuck workers are killed and retried
- **Flexible timeouts**: Specify timeout duration per code block
- **Process isolation**: Timer server runs separately from workers
- **Customizable**: Can implement custom server/client pairs
- **Localized**: Each host manages its own workers' timers

## Typical Usage Pattern
1. Start timer server alongside elastic agent
2. Configure timer client in worker processes
3. Wrap potentially stuck operations with `expires()` context
4. Server monitors and kills workers exceeding time limits
5. Agent automatically retries failed worker groups

This system ensures distributed training jobs can recover from worker hangs without manual intervention.
