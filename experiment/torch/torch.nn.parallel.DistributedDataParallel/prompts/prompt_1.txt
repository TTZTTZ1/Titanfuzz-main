**DistributedDataParallel (DDP)** is a PyTorch module for distributed training that synchronizes gradients across multiple GPUs or nodes. It wraps a model to enable data parallelism, where each process handles a subset of the data, and gradients are averaged across all processes.

### Key Features:
- **Gradient Synchronization**: Automatically performs all-reduce on gradients during backward passes.
- **Process Management**: Requires `torch.distributed.init_process_group()` to be initialized before use.
- **Performance**: Faster than `DataParallel` for single-node multi-GPU training.
- **Mixed Precision**: Supports training with mixed `fp16`/`fp32` parameters.
- **Flexible Backends**: Recommends `nccl` backend for GPU-based training for best performance.

### Basic Usage:
1. **Initialize Distributed Process Group**:
   ```python
   torch.distributed.init_process_group(backend='nccl', world_size=N)
   ```
2. **Wrap Model**:
   ```python
   model = DistributedDataParallel(model, device_ids=[rank], output_device=rank)
   ```
3. **Training Loop**: Use standard forward/backward passes; DDP handles gradient synchronization.

### Important Parameters:
- `device_ids`: List of CUDA devices for single-device modules (set to `None` for CPU/multi-device).
- `broadcast_buffers`: Synchronizes buffers (e.g., BatchNorm stats) at the start of each forward pass.
- `find_unused_parameters`: Marks unused parameters for reduction (useful for models with conditional computation).
- `static_graph`: Optimizes performance for static computation graphs.
- `bucket_cap_mb`: Controls gradient bucket size for overlapping communication with computation.

### Context Managers:
- `no_sync()`: Disables gradient synchronization within the context, useful for gradient accumulation.
- `join()`: Handles uneven inputs across processes by shadowing collective communications.

### Communication Hooks:
Allows custom gradient aggregation (e.g., compression, gossip algorithms) via `register_comm_hook`.

### Warnings:
- Ensure all processes have identical models and parameter registration order.
- Avoid modifying the model after wrapping with DDP.
- Use `forkserver` or `spawn` start method with DataLoader when using `nccl`/`gloo` backends to prevent deadlocks.

### Example:
```python
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

dist.init_process_group(backend='nccl', world_size=4)
model = DDP(model, device_ids=[local_rank])
# Standard training loop
for data in dataloader:
    loss = model(data).mean()
    loss.backward()
    optimizer.step()
```

DDP is essential for scalable multi-GPU training, providing efficient gradient synchronization and flexibility for advanced distributed training scenarios.
