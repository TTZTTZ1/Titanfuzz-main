## **Overview**
`torch.distributed` provides communication primitives for multi-process parallelism across one or more machines. It supports synchronous distributed training via `DistributedDataParallel` and offers advantages over `DataParallel` by maintaining separate optimizers per process and avoiding Python GIL limitations.

## **Key Components**

### **Backends**
- **Gloo**: Recommended for CPU training and as a fallback for GPU.
- **NCCL**: Optimized for multi-GPU training (CUDA), especially with InfiniBand.
- **XCCL**: For XPU GPU training.
- **MPI**: Optional, requires building from source.

### **Initialization**
- Use `init_process_group()` or `init_device_mesh()` to set up the process group.
- Initialization methods:
  - **TCP**: Specify IP/port and ranks.
  - **Shared File System**: Use a shared file for coordination.
  - **Environment Variables**: Default method using `MASTER_ADDR`, `MASTER_PORT`, etc.

### **Communication Primitives**
- **Point-to-Point**: `send()`, `recv()`, `isend()`, `irecv()`
- **Collectives**: 
  - `broadcast()`, `all_reduce()`, `reduce()`
  - `all_gather()`, `gather()`, `scatter()`
  - `all_to_all()`, `barrier()`
- **Object Collectives**: Support Python objects via pickling (with performance caveats).

### **Groups**
- Create sub-groups with `new_group()` for fine-grained communication.
- `DeviceMesh` manages process groups for multi-dimensional parallelism.

### **Key-Value Stores**
- `TCPStore`, `FileStore`, `HashStore` for coordination and initialization.

## **Usage Patterns**
- **Synchronous vs Asynchronous**: Most collectives support `async_op` flag for non-blocking operations.
- **Multi-GPU**: Use NCCL backend with one GPU per process.
- **Launch Utilities**: `torch.distributed.launch` and `torch.multiprocessing.spawn` simplify multi-process startup.

## **Debugging & Profiling**
- **Debugging**: Use `monitored_barrier()`, `TORCH_DISTRIBUTED_DEBUG`, and `torch.distributed.breakpoint()`.
- **Profiling**: Integrates with `torch.profiler` for performance analysis.
- **Logging**: Control via `TORCH_CPP_LOG_LEVEL` and `TORCH_DISTRIBUTED_DEBUG`.

## **Best Practices**
1. Use NCCL for GPU training, Gloo for CPU.
2. Ensure proper cleanup with `destroy_process_group()`.
3. Avoid object collectives for performance-critical paths.
4. Set network interface variables (`NCCL_SOCKET_IFNAME`/`GLOO_SOCKET_IFNAME`) if auto-detection fails.
