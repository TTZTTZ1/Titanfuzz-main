The PyTorch `torch.distributed` package enables distributed parallel training across multiple processes and machines. It provides communication primitives and supports several backends (GLOO, NCCL, MPI, XCCL) with different device capabilities. Key features include:

## **Initialization**
- Use `init_process_group()` or `init_device_mesh()` to set up the distributed environment.
- Initialization methods: TCP, shared file-system, or environment variables (default: `env://`).
- Backend selection:
  - **NCCL** for CUDA GPU training (best performance).

## **Core Concepts**
- **Process Groups**: Default group (world) or custom groups via `new_group()`.
- **DeviceMesh**: Higher-level abstraction for managing multi-dimensional device layouts and process groups.
- **Collective Operations**: Synchronous/asynchronous operations like `all_reduce`, `broadcast`, `all_gather`, etc.

- **Copy Engine Collectives**: Optimize NCCL operations by offloading data movement to GPU copy engines (requires NCCL ≥2.28 and P2P GPU access).

## **Debugging & Profiling**
- **Debugging**: Use `torch.distributed.breakpoint()`, `monitored_barrier()`, and `TORCH_DISTRIBUTED_DEBUG` environment variable.
- **Profiling**: Integrates with `torch.profiler` to profile collective communication.
- **Logging**: Control log levels via `TORCH_CPP_LOG_LEVEL` and `TORCH_DISTRIBUTED_DEBUG`.

## **Utilities**
- **Launch Scripts**: `torch.distributed.launch` (deprecated, use `torchrun`) for multi-process training.

- **Key-Value Stores**: `TCPStore`, `FileStore`, `HashStore` for inter-process coordination.

## **Best Practices**
- Ensure exclusive GPU access per process when using NCCL.
- Call `destroy_process_group()` to clean up resources.
- Use `async_op=True` for overlapping communication and computation.
- Prefer tensor collectives over object collectives for performance.

This package is essential for scalable multi-node/multi-GPU training, offering both low-level communication APIs and high-level abstractions like `DistributedDataParallel`.
