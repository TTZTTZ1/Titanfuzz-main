The `torch.distributed` package provides PyTorch support for multiprocess parallelism across multiple machines or GPUs. It enables synchronous distributed training and communication primitives for exchanging data between processes.

## Key Features

### Backends
- **Gloo**: Recommended for CPU training and as fallback for GPU
- **NCCL**: Recommended for CUDA GPU training (best performance)
- **XCCL**: For XPU GPU training
- **MPI**: Optional, requires building from source

### Initialization Methods
1. **Environment variables** (default): `MASTER_ADDR`, `MASTER_PORT`, `WORLD_SIZE`, `RANK`
2. **TCP**: Specify IP address and port
3. **Shared file system**: Use a shared file for coordination

### Core Components
- **Process Groups**: Manage communication between subsets of processes
- **DeviceMesh**: Higher-level abstraction for managing process groups across device topologies
- **Collective Operations**: `all_reduce`, `broadcast`, `all_gather`, `scatter`, etc.
- **Point-to-Point**: `send`, `recv`, `isend`, `irecv`

### Key Functions
- `init_process_group()`: Initialize distributed package
- `init_device_mesh()`: Create DeviceMesh for multi-dimensional parallelism
- `new_group()`: Create sub-groups for fine-grained communication
- `destroy_process_group()`: Clean up resources

### Distributed Data Parallel
- `DistributedDataParallel`: Wrapper for synchronous distributed training
- Advantages over `DataParallel`: Independent optimizers per process, no Python GIL thrashing

### Key-Value Stores
- **TCPStore**: Network-based store for multi-machine setups
- **FileStore**: File-based store for shared filesystems
- **HashStore**: In-memory store for single-process multi-threaded use

### Debugging Tools
- `monitored_barrier()`: Enhanced barrier with timeout detection
- `TORCH_DISTRIBUTED_DEBUG`: Environment variable for debug logging
- `breakpoint()`: Interactive debugging for single rank
- HTTP debug server for runtime diagnostics

### Launch Utilities
- `torch.distributed.launch`: Spawn multiple processes per node (deprecated in favor of `torchrun`)
- `torch.multiprocessing.spawn`: Python multiprocessing spawn utility

## Best Practices
- Use NCCL backend for CUDA GPU training
- Use Gloo for CPU training or as GPU fallback
- Always call `destroy_process_group()` for cleanup
- Set `async_op=True` for overlapping communication with computation
- Use environment variable initialization for simplicity

The package supports both synchronous and asynchronous operations, with the latter returning `Work` objects that can be waited upon or queried for completion.
