The **Elastic Agent** is the control plane in PyTorch's torchelastic framework, responsible for launching, managing, and supervising worker processes in distributed PyTorch jobs. It ensures fault tolerance by restarting workers upon failures and supports elasticity by adapting to membership changes.

### Key Responsibilities:
1. **Distributed Initialization**: Provides workers with necessary information to call `torch.distributed.init_process_group()`.
2. **Fault Tolerance**: Monitors workers and restarts the entire group if any worker fails or becomes unhealthy.
3. **Elasticity**: Reacts to membership changes (e.g., scaling) and restarts workers with updated members.

### Core Components:
- **`ElasticAgent`**: Abstract base class for agent implementations.
- **`WorkerSpec`**: Blueprint defining worker configuration (role, entrypoint, world size, etc.).
- **`WorkerGroup`**: A set of `Worker` instances managed by an agent.
- **`WorkerState`**: Tracks the state of a worker group (e.g., `HEALTHY`, `FAILED`, `STOPPED`).

### Implementations:
- **`LocalElasticAgent`**: Manages local workers on a single host, typically one agent per node. It handles spawning, monitoring, and restarting workers, with support for GPU configurations.

### Extending the Agent:
- **`SimpleElasticAgent`**: Recommended base class for custom agents, providing scaffolding for core operations like `_start_workers`, `_stop_workers`, and `_monitor_workers`.

### Additional Features:
- **Watchdog**: Optional named pipe-based timer for detecting agent/worker hangs.
- **Health Check Server**: Optional TCP/HTTP server to monitor agent liveness (enabled via environment variables).

### Usage Example:
Agents are deployed per node and manage one or more workers. They handle retries up to `max_restarts` and return a `RunResult` indicating success or failure of the worker group.
