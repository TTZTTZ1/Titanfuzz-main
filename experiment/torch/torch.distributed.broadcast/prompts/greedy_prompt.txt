The `torch.distributed` package provides PyTorch's distributed communication primitives for multi-process parallelism across multiple machines or GPUs. It supports synchronous distributed training and is essential for scaling deep learning models.

## Key Features

### Backends
- **Gloo**: Best for CPU training and Ethernet networks
- **NCCL**: Recommended for CUDA GPU training (especially with InfiniBand)
- **XCCL**: For XPU GPU training
- **MPI**: Optional backend requiring source build

### Initialization Methods
1. **Environment Variables** (default): Uses `MASTER_ADDR`, `MASTER_PORT`, `WORLD_SIZE`, `RANK`
2. **TCP**: Specify IP address and port
3. **Shared File System**: Uses a shared file for coordination

### Core Components
- **Process Groups**: Manage communication between subsets of processes
- **DeviceMesh**: Higher-level abstraction for managing multi-dimensional device layouts
- **Collective Operations**: 
  - Point-to-point: `send`, `recv`, `isend`, `irecv`
  - Collective: `all_reduce`, `broadcast`, `all_gather`, `scatter`, etc.
- **Key-Value Stores**: `TCPStore`, `FileStore`, `HashStore` for coordination

### Key Functions
- `init_process_group()`: Initialize distributed package
- `init_device_mesh()`: Create device mesh for multi-dimensional parallelism
- `new_group()`: Create sub-groups for fine-grained communication
- `destroy_process_group()`: Clean up resources

### Best Practices
1. Use NCCL backend for GPU training, Gloo for CPU
2. Set `async_op=True` for overlapping communication with computation
3. Ensure proper GPU device assignment with `torch.cuda.set_device()`
4. Clean up with `destroy_process_group()` to avoid hangs

### Debugging Tools
- `TORCH_DISTRIBUTED_DEBUG`: Enable detailed logging
- `monitored_barrier()`: Synchronization with timeout reporting
- `breakpoint()`: Debug single rank interactively
- Debug HTTP server for runtime diagnostics

The package enables efficient distributed training by allowing each process to maintain its own optimizer and Python interpreter, reducing communication overhead compared to alternatives like `DataParallel`.
