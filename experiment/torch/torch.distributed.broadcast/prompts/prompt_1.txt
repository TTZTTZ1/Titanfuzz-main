The `torch.distributed` package enables distributed parallel training across multiple processes and machines. Here's a concise summary of its key aspects:

## **Core Concepts**
- **Multi-process parallelism** across nodes/machines
- **Synchronous distributed training** via `DistributedDataParallel`
- Each process has its own Python interpreter and optimizer

## **Backends**
- **NCCL**: Recommended for CUDA GPU training (best performance)
- **Gloo**: Recommended for CPU training
- **XCCL**: For XPU GPU training
- **MPI**: Optional, requires building from source

## **Initialization Methods**
1. **Environment variables** (default: `env://`)
   - `MASTER_ADDR`, `MASTER_PORT`, `WORLD_SIZE`, `RANK`
2. **TCP**: `tcp://IP:PORT`
3. **Shared file system**: `file:///path/to/file`

## **Key Functions**
- `init_process_group()`: Initialize distributed package
- `init_device_mesh()`: Initialize DeviceMesh for multi-dimensional parallelism
- `new_group()`: Create sub-groups for fine-grained communication
- `destroy_process_group()`: Clean up resources

## **Communication Primitives**
- **Point-to-point**: `send()`, `recv()`, `isend()`, `irecv()`
- **Collectives**: 
  - `all_reduce()`, `broadcast()`, `all_gather()`
  - `reduce_scatter()`, `all_to_all()`, `barrier()`
- **Object collectives**: Work with Python objects (but have performance limitations)

## **DeviceMesh**
- Higher-level abstraction for managing process groups
- Represents multi-dimensional device layouts
- Simplifies N-dimensional parallelism setup

## **Debugging Tools**
- `monitored_barrier()`: Timeout-aware barrier with failure reporting
- `TORCH_DISTRIBUTED_DEBUG`: Environment variable for debug logging
- `breakpoint()`: Interactive debugging on specific ranks
- Debug HTTP server for runtime data collection

## **Best Practices**
- Use NCCL for GPU training, Gloo for CPU
- Call `destroy_process_group()` for cleanup
- Set `async_op=True` for overlapping communication/computation
- Use `DeviceMesh` for complex multi-dimensional parallelism
- Enable debugging flags (`TORCH_DISTRIBUTED_DEBUG`) when troubleshooting

## **Launch Utilities**
- `torch.distributed.launch`: Spawn multiple processes per node (deprecated in favor of `torchrun`)
- `torch.multiprocessing.spawn`: Python 3.4+ multiprocess spawning

The package supports both synchronous and asynchronous operations, with the latter returning `Work` objects that can be waited on or queried for completion.
