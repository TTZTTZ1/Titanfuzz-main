---

### **Core Component: `DataLoader`**
- **Purpose**: Creates an iterable over a dataset, supporting batching, shuffling, and parallel data loading.
- **Key Arguments**:
  - `dataset`: The dataset to load from (map-style or iterable-style).
  - `batch_size`: Number of samples per batch.
  - `shuffle`: Whether to reshuffle data each epoch (for map-style datasets).
  - `num_workers`: Number of subprocesses for data loading (0 = main process).
  - `collate_fn`: Function to merge samples into a batch.
  - `pin_memory`: Enables faster data transfer to GPU by using pinned memory.

---

### **Dataset Types**
1. **Map-style Datasets**:
   - Implement `__getitem__()` and `__len__()`.
   - Represent a mapping from indices/keys to data samples (e.g., reading images from disk).
   - Use `Dataset` as the base class.

2. **Iterable-style Datasets**:
   - Implement `__iter__()`.
   - Represent a stream of data (e.g., from a database or real-time logs).
   - Use `IterableDataset` as the base class.
   - **Note**: In multi-process loading, each worker gets a replica; implement sharding to avoid duplicate data.

---

### **Data Loading Order & Samplers**
- For **map-style datasets**, samplers (e.g., `RandomSampler`, `SequentialSampler`) define the order of indices.
- `shuffle=True` automatically creates a random sampler; custom samplers can be provided via the `sampler` argument.
- `batch_sampler` can be used to yield batches of indices directly.
- **Iterable-style datasets** control order via their `__iter__()` method; samplers are not applicable.

---

### **Batching**
- **Automatic batching** (default): Enabled by setting `batch_size`. Samples are collated into batches using `collate_fn`.
- **Disable batching**: Set `batch_size` and `batch_sampler` to `None`. `collate_fn` then processes individual samples.
- **Custom `collate_fn`**: Override the default to handle padding, custom data structures, or non-standard batching.

---

### **Single- vs. Multi-process Loading**
- **Single-process** (`num_workers=0`): Simple, good for debugging or small datasets.
- **Multi-process** (`num_workers>0`):
  - Speeds up data loading by using multiple worker processes.
  - **Warning**: May increase memory usage; avoid passing large Python objects to workers.
  - Platform-specific behaviors (Unix uses `fork`, Windows/MacOS uses `spawn`).
  - Use `worker_init_fn` to configure each worker (e.g., for dataset sharding).

---

### **Memory Pinning**
- Set `pin_memory=True` to accelerate host-to-GPU data transfers.
- Works automatically with Tensors; for custom batch types, implement a `pin_memory()` method.

---

### **Utilities & Samplers**
- **Dataset Utilities**: `TensorDataset`, `ConcatDataset`, `Subset`, `random_split`.
- **Samplers**: `RandomSampler`, `SequentialSampler`, `WeightedRandomSampler`, `BatchSampler`, `DistributedSampler` (for distributed training).

---

### **Key Notes**
- Use `if __name__ == '__main__':` guard when using multi-process loading on Windows/MacOS.
- For reproducibility in multi-process loading, seed workers via `worker_init_fn`.
- `len(dataloader)` is heuristic-based for `IterableDataset`.
- Avoid returning CUDA tensors directly from workers; use `pin_memory=True` instead.
