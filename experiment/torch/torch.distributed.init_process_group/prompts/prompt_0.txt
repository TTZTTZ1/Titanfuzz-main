The `torch.distributed` package provides PyTorch's distributed communication capabilities for multi‑process parallelism across multiple machines or GPUs. It supports synchronous distributed training, primarily through `DistributedDataParallel`, and offers several backends (Gloo, NCCL, MPI, XCCL) with different device and operation support.

## Key Features

- **Backends**: Gloo (CPU/GPU), NCCL (CUDA GPU), XCCL (XPU GPU), MPI (CPU/GPU with CUDA support if built). NCCL is recommended for CUDA GPU training; Gloo for CPU.
- **Initialization**: Use `init_process_group()` or `init_device_mesh()` to set up the process group. Methods include TCP, shared file‑system, and environment variables.
- **Collective Operations**: Synchronous/asynchronous collectives like `all_reduce`, `broadcast`, `all_gather`, `scatter`, `reduce_scatter`, `all_to_all`, and `barrier`.
- **Point‑to‑Point**: `send`/`recv` (synchronous) and `isend`/`irecv` (asynchronous).
- **Groups**: Create sub‑groups with `new_group()` for fine‑grained communication.
- **DeviceMesh**: Higher‑level abstraction for managing multi‑dimensional process groups (e.g., for 2D/3D parallelism).
- **Key‑Value Stores**: `TCPStore`, `FileStore`, `HashStore` for exchanging connection info.
- **Debugging**: Tools like `monitored_barrier`, `TORCH_DISTRIBUTED_DEBUG`, and `torch.distributed.breakpoint` for debugging hangs/desyncs.
- **Launch Utilities**: `torch.distributed.launch` (deprecated in favor of `torchrun`) and `torch.multiprocessing.spawn` for multi‑process startup.

## Typical Usage Pattern

1. **Initialize** the process group (e.g., via environment variables).
2. **Wrap** the model with `DistributedDataParallel`.
3. **Use** collective or point‑to‑point communication as needed.
4. **Clean up** with `destroy_process_group()`.

## Example

```python
import torch.distributed as dist

dist.init_process_group(backend='nccl', init_method='env://')
model = torch.nn.parallel.DistributedDataParallel(model, device_ids=[rank])
# ... training loop ...
dist.destroy_process_group()
```

The package is designed for SPMD (single program, multiple data) execution, where the same script runs on all ranks, and provides performance benefits over `DataParallel` by avoiding Python interpreter overhead and GIL contention.
