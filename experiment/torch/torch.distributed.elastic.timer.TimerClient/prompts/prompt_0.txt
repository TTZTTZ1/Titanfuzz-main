## Core Concept
Expiration timers allow processes to self-impose time limits on code blocks. If a worker doesn't release its timer before the deadline, the timer server can kill the stuck process, enabling the elastic agent to retry the worker.

- `configure(timer_client)`: Must be called before using timers
- `expires(after, scope=None, client=None)`: Context manager that acquires a timer expiring in `after` seconds

## Key Features
- **Process Isolation**: Timer server runs separately from workers
- **Automatic Cleanup**: Stuck workers are automatically terminated
- **Retry Mechanism**: Elastic agent can restart failed workers
- **Customizable**: Can implement custom server/client pairs by extending base classes
- **Local Scope**: Each host manages timers for its local workers

## When to Use
- Distributed training operations that might hang (e.g., collective operations, I/O)
- Ensuring training progress despite occasional worker stalls
- Building fault-tolerant distributed systems

The system provides a safety net for distributed operations, ensuring that temporary hangs don't permanently stall the entire training job.
